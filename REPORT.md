# CS2 Faceit Stats Bot — Отчёт о разработке

## Цель проекта

Разработать Telegram-бота для анализа статистики игроков в матчах Faceit по CS2. 
Faceit предоставляет множество числовых показателей после матча (K/D, ADR, HS%, MVPs, мультикиллы), но не даёт комплексной оценки вклада игрока в результат команды.

Бот преобразует сырые цифры в единую метрику — **Usefulness Score** (−2.0 до +2.0) — и вербальный вердикт (Carry / Good / Average / Weak / Throw).

---

## Этапы разработки

### Версия 0.1 — OCR-парсинг скриншотов

Начальная реализация: бот принимал скриншот матча, распознавал текст через Tesseract OCR и возвращал распознанные цифры. Подход оказался нежизнеспособным:

- OCR неустойчив к нестандартным шрифтам и сжатию
- Изменения дизайна Faceit ломали парсинг
- Решение дублировало информацию, уже доступную пользователю визуально

**Решение:** переход к работе через API.

### Версия 0.2 — Faceit API + система скоринга

Подключён Faceit API (`open.faceit.com/data/v4`). Реализованы:
- HTTP-клиент для запросов к API
- Агрегация данных матча
- Первая версия Usefulness Score

**Проблема:** скоринг использовал дискретные пороги. K/D 1.2 и 1.4 получали одинаковую оценку (1.0), так как оба попадали в категорию `good`. При этом K/D 1.19 оценивался в 0.0 (`average`). Граница между категориями была слишком резкой, что приводило к неадекватным скачкам оценки.

**Решение:** внедрена функция `_score_metric` с линейной интерполяцией между порогами. Значение метрики плавно изменяется от −2.0 до +2.0 в зависимости от близости к порогам excellent / good / average / bad.

### Версия 0.3 — Impact-категория

Добавлена категория Impact, включающая MVPs и мультикиллы (triple/quadro/penta). Ранее скор рассчитывался только по боевым метрикам.

**Проблема:** Faceit API не возвращает MVPs в статистике матча. Значение всегда равно 0, что приводило к оценке −1.0 за эту метрику даже для игроков с высоким импактом. В результате Impact-категория систематически занижала общий скор.

**Решение:** пороги MVPs изменены — значение 0 теперь соответствует нейтральной оценке (0.0), а не −1.0. Аналогичное изменение применено к multi_kills.

### Версия 0.4 — Интеграция Faceit Analyser API (эксперимент)

Получен API-ключ от Faceit Analyser (предоставлен Filipe, автор сервиса). Планировалось:
- Получать расширенную статистику по матчу (entry, trade, utility, clutch, HLTV)
- Расширить скоринг до 6 категорий

**Проблема:** Cloudflare на стороне Faceit Analyser блокирует запросы с хостинга Render (облачный IP-диапазон). На локальном окружении API работал корректно, с Render возвращал 403 Forbidden.

Предпринятые попытки:
- Изменение endpoint'ов (`/api/match/...`, `/api/stats/...`)
- Различные способы передачи ключа (query-параметр, заголовок)
- Fallback-схема: player-stats как основной, match как запасной

**Итог:** интеграция отложена. Код удалён из проекта. Для возобновления потребуется прокси-сервер или добавление IP Render в белый список Cloudflare.

### Версия 0.5 — Пересборка скоринга на доступных данных

Анализ показал, что Faceit API бесплатно предоставляет все необходимые данные:

| Данные | Категория |
|--------|-----------|
| K/D, ADR, KPR, HS% | Combat |
| MVPs, Triple/Quadro/Penta Kills | Impact |
| First Kills / First Deaths | Entry (процент выигранных открывашек) |
| KAST | Kast (процент раундов с вкладом) |

Финальная структура скора:

- Combat — 35%
- Impact — 20%
- Entry — 25%
- Kast — 20%

Никаких внешних API — только данные, которые Faceit отдаёт нативно.

## Архитектура

```
bot.py                   — точка входа, Telegram-хендлеры
faceit_client.py         — Faceit API (матчи, игроки, lifetime)
cs2space_client.py       — cs2.space (Leetify)
aggregator.py            — склейка данных из разных источников
analyzer.py              — _score_metric + compute_usefulness
formatter.py             — форматирование сообщений
parser.py                — парсинг ссылок на матчи
config.py                — конфигурация
models.py                — датаклассы
```

**Хостинг:** Render (free web service), автоматический деплой при пуше в master.

**Стек:** Python 3.11, aiogram 3, httpx.

## Usefulness Score — принцип работы

Каждая метрика (K/D, ADR, entry%, KAST, MVPs, мультикиллы) преобразуется в оценку от −2.0 до +2.0:

- значение ≥ excellent → 2.0
- значение ≥ good → 1.0–2.0 (линейно)
- значение ≥ average → 0.0–1.0 (линейно)
- значение ≥ bad → −1.0–0.0 (линейно)
- значение < bad → −2.0–−1.0

Пороги метрик:

| Метрика | excellent | good | average | bad |
|---------|-----------|------|---------|-----|
| K/D | 1.5 | 1.2 | 0.9 | 0.6 |
| ADR | 100 | 80 | 60 | 40 |
| KPR | 0.85 | 0.70 | 0.55 | 0.40 |
| HS% | 55 | 45 | 35 | 25 |
| Entry% | 75 | 60 | 40 | 20 |
| KAST% | 80 | 70 | 55 | 40 |
| MVPs | 4 | 2 | 0 | −1 |
| multi_kills | 4 | 2 | 0 | −1 |

Итоговый вердикт по сумме категорий:

- ≥ 1.5 → Carry
- ≥ 0.5 → Good
- ≥ −0.5 → Average
- ≥ −1.5 → Weak
- < −1.5 → Throw

## Тестирование

Разработано 29 тестовых сценариев, покрывающих:

- **Carry:** максимальные метрики, топ-фраггер, импакт-игрок
- **Good:** стабильная игра, entry-фрагер, саппорт с высоким KAST
- **Average:** средние показатели, пограничные значения
- **Weak:** низкие показатели, провальный entry
- **Throw:** минимальные метрики, дисконнект
- **Edge cases:** нулевые MVPs, отсутствие entry-данных, отсутствие KAST
- **Consistency:** монотонность оценки, компенсация слабых категорий сильными

Все тесты проходят.
